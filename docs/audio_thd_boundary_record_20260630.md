# 音频 THD 指标边界确认记录（2026-06-30）

## 1. 我为什么要补这份记录

旧项目整理材料里经常会出现一句话：

- “音频总失真率控制在 `< 5%`”

这个数字很适合写进简历，但我重新审项目时发现一个问题：

- 当前工程里虽然已经把音频采集、AAC 编码、MP4、RTSP 都跑通了
- 但并没有现成的 `THD < 5%` 实测报告

如果我直接把这个数字写成“我已经实现”，风险很大。  
所以我这次做的不是强行凑一个好看的值，而是先把**什么条件下可以测 THD，什么条件下不能乱测**这件事讲清楚。

---

## 2. 我这次用了什么样本

我先复用了工程资料里已有的音频样本：

- `第二期源码/11-3_音频文件格式/pcm_to_wav/input.pcm`
- `第二期源码/11-3_音频文件格式/pcm_to_wav/output.wav`

我先确认了 `output.wav` 的基本参数：

- `44100Hz`
- `2` 声道
- `16bit`
- 总时长约 `159.18s`

这一步的目的不是证明当前板端音频参数，而是先拿已有样本验证分析工具链能不能工作。

---

## 3. 我新增的分析脚本

我补了一个脚本：

- `tools/analyze_audio_quality.py`

这个脚本支持两种输入：

1. `WAV`
2. 原始 `PCM`

它输出的内容包括：

- 采样率
- 声道数
- 位宽
- 时长
- 峰值
- RMS
- 主频
- 粗略 THD 估计
- 一条适用性说明

这里我专门加了一条保护逻辑：

- 只有输入接近“单一主频的近似正弦波”时
- THD 这个值才有资格被当成失真指标去讨论

如果输入本身是语音、音乐或复杂宽带信号，脚本会明确提醒：

- 这个 THD 值**不适合拿来当简历指标**

---

## 4. 我这次跑出来的结果

### 4.1 WAV 样本

运行：

```bash
python tools/analyze_audio_quality.py --wav 第二期源码/11-3_音频文件格式/pcm_to_wav/output.wav
```

结果核心字段：

- `sample_rate = 44100`
- `channels = 2`
- `duration_sec = 159.18`
- `dominant_freq_hz ≈ 242.25`
- `thd_percent ≈ 35.04`

### 4.2 PCM 样本

运行：

```bash
python tools/analyze_audio_quality.py \
  --pcm 第二期源码/11-3_音频文件格式/pcm_to_wav/input.pcm \
  --sample-rate 44100 \
  --channels 2 \
  --sample-width 2
```

结果和 WAV 基本一致：

- `dominant_freq_hz ≈ 242.25`
- `thd_percent ≈ 35.04`

这说明：

- `input.pcm` 和 `output.wav` 本质上是同一段音频内容
- 脚本链路是通的

## 5. 我为什么没有把 35.04% 写进简历

因为这一步实验真正告诉我的不是“音频很差”，而是：

- 这段样本本身并不是标准的单频正弦波测试信号
- 用它直接算 THD，不适合作为正式失真指标

换句话说：

- 这次实验的价值在于**确认测量边界**
- 而不是为了硬凑出一个数字

如果我把这 `35.04%` 写进简历，那其实是在误用 THD 概念。

### 6.2 当前我不能乱讲的内容

因为还缺严格的标准测试输入和闭环测量。

### 6.3 真正要做严谨 THD 实验，需要什么

如果后面我要把这项指标做实，正确流程应该是：

1. 准备标准 `1kHz` 正弦波输入
2. 明确采集链路参数
3. 采板端采集/编码/解码后的输出
4. 对基波和 2~5 次谐波做频谱分析
5. 再给出 THD 百分比

只有这套闭环跑完，`< 5%` 这种数字才讲得住。
