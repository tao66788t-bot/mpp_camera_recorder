这部分是当前项目里最稳、最适合在简历里直接讲的结果。

这一条是当前项目里最适合保留的“效果提升”类数字。

## 3. 当前发现与旧文档不一致的量化项

### 3.1 音频不是 44.1kHz / 双声道 / 128kbps

很多旧整理文档里写的是：

- `44.1kHz`
- `双声道`
- `AAC 128kbps`

但当前项目真实代码不是这样。

当前代码实际配置：

- `sample_rate = 16000`
- `channels = 1`
- `bit_width = 16`
- `aenc_bitrate = 16000`

证据来源：

- `src/main.c`
- `src/audio_module.c`
- `src/mux_module.c`
- `docs/audio_capture_validation_20260628.md`

所以面试时不能直接复述旧模板里的：

- `44.1kHz`
- `双声道`
- `128kbps`

否则一旦被追问代码细节，就会和当前工程对不上。

### 3.2 当前 MP4 封装并不是“我自己完整手写了一个 MUX 文件格式器”

当前工程使用的是全志 MPP 的 `AW_MPI_MUX_*` 接口：

- `AW_MPI_MUX_CreateGrp`
- `AW_MPI_MUX_CreateChn`
- `AW_MPI_MUX_SendVideoStream`
- `AW_MPI_MUX_SendAudioStream`

也就是说：

- 我做的是“基于全志 MPP 的多线程流路由与封装接入”
- 不是从零自己手撸 MP4 box 写盘器

这和学长梳理文档里“大篇幅拆解 MP4 box”的讲法不完全等价。

更准确的说法应该是：

- 我研究了 MP4 封装结构和 MUX 工作机制
- 结合生产者/消费者式的码流路由，把视频流和音频流同时送入 MUX
- 让板端稳定生成可播放的 MP4 文件

### 3.3 当前多线程优化不是“完整双消费者共享队列”实现

当前代码真实结构更接近：

- `RTSP server thread`
- `stream_router_thread`
- `stream_router_thread` 里拉取 `VENC_GetStream / AENC_GetStream`
- 同一个 router 线程把流同时发给：
  - `MUX`
  - `RTSP`

证据来源：

- `src/rtsp_module.c`
- `include/rtsp_module.h`

这说明：

- 当前确实是“多线程”
- 但不是旧文档里那种严格意义上的
  - `producer -> shared queue -> consumer1/consumer2`
  的完整共享节点引用计数实现

因此：

- “有生产者消费者设计思路”可以讲
- “当前版本代码就是完整双消费者共享队列实现”不能直接讲死

## 4. 当前仍然证据不足的量化项

下面这些数字，目前更多来自旧文档模板、学长项目梳理或面试草稿，不属于当前工程已经用证据闭环的结果。

### 4.1 色彩还原度提升约 20%

旧文档里给出了一个可能的测量方法：

- 用色卡
- 用 Lab / ΔE
- 例如 `8.5 -> 6.8`
- 改善率约 `20%`

但当前项目里还没有发现：

- 真正拍下来的色卡样本
- 当前板端图像的优化前后原始对照
- 自动化 ΔE 计算脚本
- 当前这块板子的测试报告

所以这一项目前只能算“可设计的测量方案”，不能算“已实证”。

### 4.2 音频总失真率 < 5%

旧材料里把失真分解成了：

- 采集失真
- 编码失真
- 解码失真
- 播放失真
- 其他失真

然后叠加成 `< 5%`

但当前工程没有以下实证链路：

- 正弦波注入
- FFT / THD 实测
- 当前板子的解码播放测量数据

所以这项目前不能直接说“我已经测到 < 5%”，最多只能说：

- “我整理过 THD 的测量方法”
- “如果要做严格量化，需要补 1kHz 正弦波 + FFT 频谱实验”

### 4.3 MP4 封装速度 +40%，I/O 调用次数 -60%，CPU 65% -> 85%

这组数据在旧文档里出现很多次，但当前工程里没有发现：

- 单线程和多线程两个独立可切换版本
- 一致输入下的性能压测脚本
- 真正采样得到的 fps / CPU / I/O 次数实验日志

并且当前实现的“多线程”也不是旧文档里的完整双消费者队列版本。

所以这组数字目前不能算当前工程已经真实落地。

### 4.4 音视频同步偏差 < 30ms

旧文档里提了 `< 30ms`，但当前项目还缺：

- 精确的音视频 PTS 对齐统计脚本
- 当前 MP4 样本或 RTSP 播放过程里的量化同步误差

所以这项目前也还是“可讲方法论”，不是“已经有当前工程实测值”。

## 6. 后续补实验优先级

### 第一优先级：把当前工程已有能力再自动汇总

可立即补齐：

- MP4 音视频轨道信息自动提取
- MP4 视频 CV 指标自动提取
- 当前音频参数自动汇总

这类工作成本低，而且能直接增强“当前项目真实度”。

### 第二优先级：补色彩还原度 20% 的测量闭环

需要新增：

- 色卡拍摄样本
- 优化前后图像对照
- ΔE 计算脚本

只有这一步做完，`20% 色彩还原提升` 才能真正从“说法”变成“证据”。

### 第三优先级：补多线程封装性能实验

需要先做结构准备：

- 保留当前 router 版本
- 再做一个更接近“单线程/弱并发”的对照版本
- 统计 fps / CPU / I/O 次数

只有这样，`+40%` 这类加速数字才讲得住。

### 第四优先级：补音频 THD 闭环

需要外部实验条件：

- 1kHz 正弦波输入
- FFT 分析
- 采集/编码/解码/播放链路测量

这一步实验成本最高，不建议先做。
