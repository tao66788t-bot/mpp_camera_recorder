# LCD 预览异常排查 —— 面试讲述稿

## 30 秒版本（电梯 pitch）

> 项目里录像正常但 LCD 预览发白过曝。我按链路拆分成两个独立问题，写了一个诊断抓帧工具把"看起来白"变成"VI2 的 YUV 数据在哪一步出了偏差"，结合 SDK sample 分析了色彩空间和窗口策略参数的物理含义，定位到 `color_space` 和 `use_current_win` 两个配置项。改完之后 LCD 恢复正常。整个过程靠的是方法论，不是靠试参数。

---

## STAR 完整版（3-5 分钟面试回答）

### S: Situation 背景

全志 V853 平台 + GC2053 摄像头 + 480×800 LCD。需要同时实现两条链路：

| 链路 | 路径 | 用途 |
|------|------|------|
| 录像 | VI0(1920×1088 NV21) → VENC H.265 → MUX → MP4 | 存 SD 卡 |
| 预览 | VI2(480×800 NV12) → VO → LCD | 实时显示 |

两条链路共用同一个 sensor 但走不同的 VI 通道，各自有独立的像素格式和色彩空间配置。

### T: Task 遇到的问题

**MP4 录像正常，但 LCD 预览画面发白、对比度差、局部过曝。** 一眼看去像 ISP 曝光问题，但录像正常说明 sensor 和 ISP 主链路没有完全失效——问题一定出在预览链路的某个环节。

### A: Action 排查过程——五个关键步骤

**第一步：链路拆分，缩小范围。**

我没有一上来就怀疑 ISP 或者 AE。录像正常意味着 sensor 输出、ISP 处理、编码链路都是通的。问题范围直接锁定在 VI2 → VO → LCD 这一段。这一步把排查空间从"整个系统几百个参数"缩减到"预览链路的十几个参数"。

**第二步：写诊断工具，把主观感受变成客观数据。**

"画面发白"在工程上没法精确 debug。我专门写了一个 `--diag-vi-dump` 命令行模式，启动后分别抓取 VI0 和 VI2 的一帧原始 YUV 数据，保存到文件。在 PC 上打开 YUV 文件逐通道分析亮度分量：

- 发现 VI2 的 Y 平面整体偏亮，直方图集中在高亮度区间
- 对比 VI0 的 YUV 数据则是正常的

这一步直接排除了 VO/LCD 渲染层的问题——数据在 VI2 输出时就已经异常了。

**第三步：研究 SDK sample，理解每个参数的含义。**

确认问题在 VI2 之后，我没有随便改参数。我研究了全志 MPP SDK 自带的 `sample_virvi2vo`、`sample_vo` 等示例代码，重点关注它们对 VI 预览链路的配置方式。同时查阅了 V4L2 规范中色彩空间相关的定义。

**第四步：定位到两个关键参数。**

通过分析诊断数据和对照 SDK 参考实现，我发现 VI2 有两个配置存在问题：

| 参数 | 问题值 | 正确值 | 物理含义 |
|------|--------|--------|----------|
| `color_space` | `REC709` | `JPEG` | 量化范围不同 |
| `use_current_win` | `1` | `0` | 窗口复用策略 |

这两个值之所以有问题，是因为我早期参照某个 sample 套用了它们的配置，但那个 sample 是单路 VI 场景，而我的项目是双路 VI 共用 sensor，上下文完全不同。

**第五步：理解原理，而不是背配置。**

我没有停留在"改这两个值就好了"。我深入理解了为什么：

- **`V4L2_COLORSPACE_JPEG` vs `V4L2_COLORSPACE_REC709`**：JPEG 色彩空间使用全范围量化（0-255），REC709 使用限制范围（16-235）。全志 VO 模块内部期望 JPEG 范围的数据，当输入是 REC709 范围时，硬件会把 16-235 映射到 0-255，导致暗部不够暗、亮部不够亮——这就是"发白发灰"的物理原因。
- **`use_current_win`**：当多路 VI 共享同一个 sensor 输出时，如果开启此标志，VI 通道会尝试复用 sensor 当前的窗口配置。但 VI2 需要 480×800 的裁剪窗口，VI0 需要 1920×1088，窗口策略冲突会导致 ISP 输出的缩放异常。

**第六步：最小化修复 + 闭环验证。**

我只改了 VI2 的这两个参数，VI0/VENC/VO/ISP 全部不动——不引入新变量。每次修改后严格走完整验证流程：重编译 → 推固件 → reboot 板子 → 等 ADB 恢复 → 单实例启动 → 确认 LCD 画面 → 确认 MP4 录像不受影响。确保结果可复现。

### R: Result 结果

- LCD 480×800 全屏预览恢复正常
- MP4 录像功能零影响
- 确认根因是预览链路配置参数，不是 ISP/sensor 问题

## 面试中的关键表达

### 1. "我把主观问题变成了客观数据"

"画面发白"没法 debug。我写了 `--diag-vi-dump` 抓帧工具，把问题转化为"VI2 的 Y 平面亮度分布在第几个处理阶段出现了异常"。**能写诊断工具来定位问题，和一个参数一个参数去试，是两个层次的工作方式。**

### 2. "我不仅知道怎么改，还知道为什么"

- `COLORSPACE_JPEG`（0-255 全范围）vs `REC709`（16-235 限制范围）→ 理解量化范围差异如何导致对比度压缩
- `use_current_win` → 理解多路 VI 共享 sensor 时的窗口策略

**换一个 sensor、换一个 SoC，这套分析方法一样能用**——因为我知道色彩空间和窗口策略在不同平台上的通用语义。

### 3. "我交叉验证了多条证据链"

不是看到一篇 blog 说"设 JPEG 就好了"就照做。我拿到诊断数据（YUV 分析）、SDK 参考代码、V4L2 规范定义三条独立证据，结论能相互印证。**单一来源的判断可能是巧合，三条独立线索指向同一个结论才是定位。**

### 4. "一次只改一个变量，结果必须可复现"

修复时只动 VI2 两个参数，不动其他任何模块。每次验证走完整流程（编译→推送→重启→启动→确认），不是"重启一下好像好了"。**可复现的修复才是真正的修复。**

## 我在这件事上希望面试官看到的三点

1. **工程基础设施意识**：出了问题不是直接改参数，而是先花时间写诊断工具。这个习惯在大项目里极其重要。

2. **理解原理而非记忆配置**：我能解释 `COLORSPACE_JPEG` 和 `REC709` 的量化范围差异，理解 `use_current_win` 在多路 VI 下的行为。这些知识不依赖一个具体平台。

3. **闭环验证的方法论**：链路拆分 → 抓帧诊断 → 研究 SDK 文档 → 单变量修复 → 完整流程验证。每一步都是可解释、可复现的。

## 一句话总结

> 录像正常、预览异常 → 按链路拆分、写抓帧工具逐段隔离 → 研究 SDK sample 和 V4L2 规范理解参数语义 → 定位到 `color_space` 和 `use_current_win` 两个配置 → 知其所以然地修复 → 闭环验证。整个过程靠的是工程方法论，不是靠试参数，也不是靠有现成答案。
