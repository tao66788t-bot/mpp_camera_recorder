# IPC 项目最终投递讲稿（2026-06-30）

## 1. 自我介绍里怎么讲这段项目

如果我在自我介绍里只用 30 到 40 秒讲这个项目，我会这么说：

> 我做过一个跑在全志 V853 上的 IPC 固件项目，核心是基于全志 MPP 打通 `GC2053` 的采集、LCD 本地预览、H.265 MP4 录制、RTSP 音视频推流和端侧检测这几条主链路。  
> 这项目里我重点解决了两个问题：一个是 LCD 预览链路偏色，最后通过抓帧和离线量化确认 `VI2 NV12 -> VO` 是正确路径；另一个是 H.265 码流波动，我对照 SDK sample 补齐 QP、VBV 和 RC 参数后，把 MP4 帧大小离散系数 `CV` 从 `143.70%` 压到了 `92.70%`。  
> 我后来还专门做了一轮量化项审计，把旧文档里的模板数字和当前工程真实结果分开，只保留能被代码、日志、样本和脚本证明的部分。

## 2. 简历投递版 1 分钟讲法

> 这个项目本质上是一个嵌入式音视频系统集成项目。  
> 我在全志 V853 板子上完成了 `VI0 -> VENC -> MUX -> MP4` 和 `VI2 -> VO -> LCD` 两条视频链路，同时把 `AI -> AENC -> AAC` 音频链路并到 MP4 和 RTSP 里。最终稳定跑通的是：`480x800` LCD 预览、`1920x1088@20fps` H.265 编码录制、RTSP `554` 端口音视频推流，以及 `480x288 NV12` 的人脸检测模型。  
> 这个项目里我不是只把功能堆起来，而是专门处理了两个比较有代表性的问题。第一个是 LCD 偏色，我通过拆链路、抓 `VI2` 原始帧、对照 SDK sample 和做离线图像一致性分析，证明修复后预览图和正确参考图是像素级一致的。第二个是 H.265 码流波动，我通过补齐 QP、VBV 和帧率控制参数，把 MP4 帧大小离散系数从 `143.70%` 优化到 `92.70%`。  
> 另外我还专门清理过简历里的量化项，像 `44.1kHz / 128kbps / THD<5% / 色彩提升20%` 这种旧模板数字，我没有直接照搬，而是按当前工程真实实现重新核实过。

## 3. 面试官问“你最有代表性的贡献是什么”时怎么讲

### 讲法 A：偏工程定位能力

> 我觉得最有代表性的贡献是把 LCD 问题从“看起来发白发黄”这种主观现象，变成了可验证的链路问题。  
> 当时 MP4 和 RTSP 都正常，只有 LCD 异常，所以我没有一上来就怀疑 ISP 全局有问题，而是先把预览链路和编码链路拆开。后面我抓了 `VI2` 的原始帧，对照全志 SDK sample 和 V4L2 色彩空间定义，定位到 `VI2` 必须走 `NV12`，而不是沿用录像链路的 `NV21`。最后我还补了离线图像量化，修复图对参考图的 `MAE/RMSE` 都是 `0.000`。  
> 这件事比较能体现我不是只会改参数，而是会先拆问题、找证据、再做最小修复。

### 讲法 B：偏优化闭环能力

> 如果从量化优化角度讲，我最完整的一项工作是 H.265 码流稳定性优化。  
> 我一开始看日志就发现 H.265 CBR 的很多参数虽然结构体里有，但实际运行时没被稳定下发，甚至出现过 `QP=0`、`VBV=0` 的情况。我对照全志 sample 后补齐了 QP 范围、VBV 和 RC 帧率参数，然后不是只看日志，而是录板端 MP4，用脚本统计每帧大小的离散系数，最后把 `CV` 从 `143.70%` 压到了 `92.70%`。  
> 这一项我后面面试里很愿意展开，因为它是完整的发现问题、改参数、拿样本、跑分析、拿结果的闭环。

## 5. 面试官问“那你对 THD 和色彩 20% 这些数字后来做了什么”时怎么讲

> 我后来没有把它们直接删掉不管，而是继续把实验底座往前补。  
> 音频这边，我补了标准正弦波生成脚本和音频质量分析脚本。现在我已经能自己生成标准 `1kHz` WAV，并且脚本可以稳定识别到 `1000Hz` 主频，说明后面只差真实板端闭环样本，不差方法和工具。  
> 色彩这边，我补了 `Lab / ΔE` 分析脚本和 ROI 模板。现在我已经能按色块计算平均 RGB、转 `Lab`，再算 `ΔE76`。虽然手头还没有正式色卡实拍样本，所以我不会把 `20%` 直接写死，但实验方法和工具已经准备好了。  
> 也就是说，我不是简单说“这项没做”，而是把它推进到了“只差标准样本就能正式闭环”的状态。
