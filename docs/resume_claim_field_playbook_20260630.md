## 2. 实验前统一前提

### 2.1 固件基线

统一使用当前可工作主线或明确备份版本，建议优先：

- `deploy_full_verified_binary_remote.sh`

如果要先恢复到一个已知稳定起点，可参考：

- `restore_full_verified_20260628.ps1`
- `backup_ipc_firmware_full_verified_20260628`

### 2.2 部署硬规则

继续严格遵守项目里已经反复验证过的流程：

1. `adb push`
2. `adb shell reboot`
3. `sleep 15`
4. `adb kill-server && adb start-server && sleep 2`
5. 再次 `adb push`
6. 单实例启动

原因：

- 这个板子硬件状态残留明显
- 不完整 reboot 很容易污染实验

### 2.3 每次实验都要留档的最小材料

每跑一轮实验，至少保留下列 4 类证据：

1. 当前版本说明
   - 用哪个二进制
   - 用哪个脚本启动

2. 原始样本文件
   - `mp4 / wav / pcm / png / jpg / log`

3. 分析脚本输出
   - JSON 或控制台输出原文

4. 第一人称实验记录
   - 做了什么
   - 遇到什么问题
   - 最终结果能不能写进简历

## 3. Claim A：`THD < 5%`

## 3.1 当前真实状态

现在已经有：

- `tools/generate_sine_wave.py`
- `tools/analyze_audio_quality.py`

还没有：

- 标准正弦波真正经过板端采集/编码链路后的闭环样本

### 3.2 现场实验目标

最小目标：

- 证明当前 `AI -> AENC(AAC) -> MP4/RTSP` 链路对标准单频输入的失真可被量化

进阶目标：

- 得到一个真实可引用的 `thd_percent`

### 3.3 样本准备

在本地生成标准输入：

```powershell
python .\tools\generate_sine_wave.py --output .\thd_1khz.wav --freq 1000 --sample-rate 16000 --duration 10 --channels 1 --amplitude 0.5
```

建议额外生成两份控制样本：

- `500Hz`
- `2kHz`

目的是排除只在单一点上“恰好好看”的结果。

### 3.4 现场执行方式

当前工程没有完整 `AO/Lineout` 播放闭环，所以这里不做“全播放链路 THD”，而做：

- **采集/编码侧音频质量实验**

推荐现场方式：

1. 用手机/电脑外放标准 `1kHz` 正弦波
2. 让板端 MIC 在相对固定距离采集
3. 启动当前固件并录出 MP4
4. 从 MP4 中取出音频样本，再做分析

### 3.5 需要保留的证据

至少保留：

- 正弦波输入文件 `thd_1khz.wav`
- 板端录出来的 MP4
- 从 MP4 导出的音频文件或 PCM
- `analyze_audio_quality.py` 输出

### 3.6 判定标准

可以形成简历结论的前提：

1. 主频稳定落在目标频点附近
2. `dominant_ratio` 足够高
3. `analysis_note` 不再是“宽带音频不适合测 THD”
4. 多次样本结果一致

如果还出现：

- 环境噪声过大
- 主频不稳
- `dominant_ratio` 很低

那就只能写成：

- “实验方法已建立，但当前现场样本还不满足正式 THD 评估条件”

## 4. Claim B：色彩提升 `20%`

## 4.1 当前真实状态

现在已经有：

- `tools/analyze_color_delta_e.py`
- `tools/color_patches_template.json`
- LCD 修复前后的诊断图像量化

还没有：

- 标准色卡实拍样本

### 4.2 现场实验目标

最小目标：

- 把“颜色看起来更对了”升级成 `ΔE` 可计算结论

进阶目标：

- 得到“平均 `ΔE` 改善比例”

### 4.3 现场样本要求

至少需要拍两组图：

1. 修复前版本
2. 修复后版本

并且两组都要满足：

- 同一块标准色卡
- 同一场景
- 同一光照
- 尽量同一拍摄角度

### 4.4 现场执行流程

1. 先部署“修复前”版本，拍色卡
2. 再部署“修复后”版本，拍同一色卡
3. 用 `color_patches_template.json` 填色块 ROI
4. 跑：

```powershell
python .\tools\analyze_color_delta_e.py --image .\after.jpg --compare-image .\before.jpg --patches .\tools\color_patches_template.json
```

### 4.5 需要保留的证据

- 两张原始实拍图
- ROI 模板
- 脚本输出 JSON
- 最终平均 `ΔE` 比较结果

### 4.6 判定标准

只有在下面条件同时成立时，才考虑写“提升百分比”：

1. 色卡样本真实可追溯
2. ROI 标注明确
3. 平均 `ΔE` 确实改善
4. 改善比例多次实验稳定

否则最多写成：

- “用 `ΔE` 方法验证修复后色彩一致性更接近参考”

## 5. Claim C：`AV sync < 30ms`

## 5.1 当前真实状态

现在已经有：

- `tools/analyze_mp4_av_sync.py`
- 容器级粗对齐结果

还没有：

- 毫秒级受控事件实验

### 5.2 现场实验目标

最小目标：

- 把“粗粒度没有明显漂移”升级到“事件级误差可估”

进阶目标：

- 得到可以支撑 `< 30ms` 的事件级统计

### 5.3 推荐实验事件

优先使用强同步事件：

1. 拍手
2. 闪光 + 蜂鸣
3. 手机屏幕闪白 + 同时播放短 beep

原因：

- 视频里容易定位帧
- 音频里容易定位波峰

### 5.4 现场执行流程

1. 启动当前完整固件
2. 对着摄像头做 5 次左右同步事件
3. 录成 MP4
4. 先用 `analyze_mp4_av_sync.py` 做粗过滤
5. 再人工或脚本逐事件对齐：
   - 视频帧出现时刻
   - 音频峰值时刻

### 5.5 当前限制

当前仓库里还没有逐帧事件对齐脚本，所以这一步现场第一次很可能需要：

- 先保留原始 MP4
- 后续再补更细粒度解析工具

### 5.6 判定标准

只有在：

- 多次事件都稳定落在目标阈值内

才考虑写：

- `AV sync < 30ms`

否则当前更安全的表述仍然是：

- “音视频轨在容器级已基本对齐，无明显漂移”

## 6. Claim D：`封装 +40% / I/O -60%`

## 6.1 当前真实状态

现在已经有：

- `tools/analyze_router_log_metrics.py`
- 多份真实历史 router 日志
- 当前结构下的粗 `fps/fail ratio`

还没有：

- 同条件可比的基线版本
- CPU / I/O 正式对照结果

### 6.2 现场实验目标

最小目标：

- 基于当前结构拿到统一口径的吞吐/失败率/CPU 数据

进阶目标：

- 在“基线版 vs 当前版”之间形成百分比对照

### 6.3 版本准备

至少准备两套版本：

1. 当前 router 版本
2. 更弱并发或更简单分发版本

如果现场没有第二套版本，就不要强做 `+40% / -60%` 百分比。

### 6.4 统一采样条件

建议每轮固定：

- 运行 60 秒
- LCD + MP4 + RTSP + 音频全部开启
- `1920x1088@20fps`
- H.265 `2Mbps`

### 6.5 现场采样内容

每轮至少留：

1. router 日志
2. `ps/top` 的 CPU 采样
3. 如环境允许，再补：
   - `strace`
   - 系统调用统计

### 6.6 现场分析流程

1. 跑固件
2. 保存控制台日志
3. 用：

```powershell
python .\tools\analyze_router_log_metrics.py .\qp_limited_console.log
```

4. 整理两版本对照：
   - `estimated_fps_from_pts`
   - `video_fail_ratio`
   - CPU
   - I/O

### 6.7 判定标准

只有在：

- 两版本输入一致
- 统计口径一致
- 至少重复 2~3 次结果稳定

时，才考虑写：

- `+40%`
- `-60%`

否则最多写成：

- “当前多线程路由版本已具备可统计的吞吐与 fail ratio，对照实验框架已建立”

## 7. 推荐现场执行顺序

如果现场时间有限，我建议按下面顺序跑：

1. 性能类粗对照
   - 因为当前工具最全，最容易出结果

2. `AV sync`
   - 因为只需要 MP4 样本和事件动作

3. `THD`
   - 需要更干净的现场音频条件

4. `ΔE`
   - 需要标准色卡和更稳定的拍摄条件
