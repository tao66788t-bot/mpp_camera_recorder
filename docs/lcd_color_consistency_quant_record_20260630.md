# LCD 预览色彩一致性量化记录（2026-06-30）

## 1. 这次为什么要补这个实验

前面我已经把 LCD 预览异常的问题定位清楚了：

- 录像链路正常
- RTSP 链路正常
- 异常集中在 `VI2 -> VO` 这一条预览链路
- 最终修复点和 `NV12 / NV21`、`color_space`、`use_current_win` 这些参数有关

但是如果面试里只说“我把 LCD 调好了”，这个说法还是偏感性。  
所以我又补了一步：直接拿项目里现成的诊断抓帧图片，做一次离线量化对比，证明修复后的预览图确实更接近正确参考图。

## 3. 我新增的量化工具

我补了一个脚本：

- `tools/analyze_lcd_color_consistency.py`

这个脚本做的事情很简单：

- 把参考图和候选图统一到同一尺寸
- 逐像素比较 RGB 差异
- 输出：
  - `MAE`（平均绝对误差）
  - `RMSE`（均方根误差）
  - 图像平均 RGB
  - 图像 RGB 标准差

我这次执行的命令是：

```bash
python tools/analyze_lcd_color_consistency.py \
  --reference diag_vi2_lcdfixed_as_nv12.png \
  --candidates \
    diag_vi2_lcdfixed_current.png \
    diag_vi2_lcdfixed_as_nv21.png \
    diag_vi2_current.png
```

## 4. 实测结果

结果如下：

### 4.1 当前修复图 vs NV12 参考图

- 文件：`diag_vi2_lcdfixed_current.png`
- `mae_rgb = 0.000`
- `rmse_rgb = 0.000`

这说明：

- 当前保留下来的 LCD 正常版本
- 和正确的 NV12 参考图在这组样本里是完全一致的

### 4.2 NV21 解释结果 vs NV12 参考图

- 文件：`diag_vi2_lcdfixed_as_nv21.png`
- `mae_rgb = 15.034`
- `rmse_rgb = 27.664`

这说明：

- 如果把同一份数据按 `NV21` 去解释
- 颜色会明显偏离正确结果

### 4.3 异常旧图 vs NV12 参考图

- 文件：`diag_vi2_current.png`
- `mae_rgb = 69.199`
- `rmse_rgb = 80.972`

这说明：

- 异常旧图和正确预览图之间的差异远大于 `NV21` 那一组
- 说明之前 LCD “发白/发灰/偏色”不是错觉，而是像素解释链路确实已经偏掉了

---

## 5. 我怎么理解这组结果

这组结果给了我两个比较扎实的结论。

### 5.1 修复后的 LCD 图不是“主观上感觉好一点”

因为：

- 当前修复图和 NV12 正确参考图的误差是 `0`
- 不是“差不多”
- 是在这组离线样本上完全一致

所以我后面可以很明确地说：

- LCD 恢复正常不是靠运气
- 是因为预览链路回到了正确的数据格式和色彩空间解释方式

### 5.2 `NV12` 和 `NV21` 在这条链路里确实不能混

这也和我前面在项目里总结过的硬规则一致：

- 录像链路 `VI0` 用 `NV21`
- LCD 预览链路 `VI2` 用 `NV12`

如果这两个格式在预览链路里混掉，颜色就会明显偏掉。

- 用定量方式证明修复后的预览图和正确 NV12 参考结果一致
- 同时证明了 `NV12 / NV21` 的混用会带来明显颜色偏差

这类说法既真实，也能和我当前项目代码、诊断图、分析脚本完全对上。

## 7. 当前边界

这次量化做的是：

- **一致性对比**

不是：

- **标准色卡 ΔE 标定**

所以我现在能严谨地说的是：

- “修复后的 LCD 结果与正确参考图一致”
- “错误格式解释会产生明显颜色偏差”

但还不能直接把它上升成：

- “色彩还原度提升 20%”

如果后面要把 `20%` 这个数字也做实，还需要补：

- 色卡实拍样本
- 色块 ROI
- Lab / ΔE 自动计算
