## 1. 我为什么要专门补这份总记录

我在整理这段全志 V853 IPC 项目经历时，发现一个很现实的问题：

- 项目里已经积累了很多文档
- 里面有很多看起来很漂亮的数字
- 但这些数字并不都和我当前这份工程完全一致

如果我不把这些数字一条条重新做实，后面面试时很容易出现一种情况：

- 简历写的是一套
- 当前代码和样本证明出来的是另一套

所以我这次做的，不是继续润色项目描述，而是反过来从证据出发，把这些量化项按“已经做实 / 需要纠偏 / 还没闭环”重新梳理了一遍。

## 4. 我怎么把 LCD 偏色问题做成了可量化的结果

LCD 预览问题是我这段项目里最值得讲的问题之一。

因为一开始这个问题特别容易被误判成：

- ISP context 丢了
- AE 有问题
- sensor 参数错了

但我后来通过拆链路发现：

- MP4 正常
- RTSP 正常
- 只有 LCD 异常

这说明问题不在全局 ISP，而更像集中在 `VI2 -> VO` 这一条预览链路。

我后面做了几步：

1. 先把录像链路和预览链路分开
2. 用诊断工具抓 `VI2` 的原始数据
3. 研究全志 SDK sample 和 V4L2 色彩空间定义
4. 确认 `VI2` 预览链路必须用 `NV12`
5. 确认 `VI0` 录像链路走 `NV21`

为了不让这个问题停留在“看起来正常了”，我又补了一步离线量化。

我写了 `analyze_lcd_color_consistency.py`，把以下几张图做逐像素对比：

- 正确参考图：`diag_vi2_lcdfixed_as_nv12.png`
- 当前修复图：`diag_vi2_lcdfixed_current.png`
- 错误格式解释图：`diag_vi2_lcdfixed_as_nv21.png`
- 旧异常图：`diag_vi2_current.png`

结果是：

- 当前修复图 vs 正确参考图：
  - `MAE = 0.000`
  - `RMSE = 0.000`
- 错误 `NV21` 解释图：
  - `MAE = 15.034`
- 旧异常图：
  - `MAE = 69.199`

这说明我最终修好的 LCD 版本，不是主观上“感觉差不多”，而是和正确参考结果在这组样本上完全一致。

## 5. 我怎么把 H.265 码流稳定性做成了真正的量化闭环

这是我目前手里最完整的一项优化闭环。

我一开始看日志就发现一个问题：

- H.265 CBR 的很多关键参数虽然结构体里有
- 但实际运行时并没有被稳定下发
- 甚至出现过：
  - `MaxQp=0`
  - `MinQp=0`
  - `VBV BufSize=0`
  - `ThreshSize=0`

我对照全志 SDK sample 后，把这几类参数补齐了：

1. H.265 CBR 的 QP 范围
2. `key_frame_interval`
3. `VBV BufSize / ThreshSize`
4. RC 层的源帧率和目标帧率

然后我做的不是只看日志，而是：

1. 板端录一段 MP4
2. 拉回主机
3. 用离线脚本 `analyze_mp4_cv.py` 统计每帧大小的离散系数

最终结果是：

- 优化前：`CV = 143.70%`
- 优化后：`CV = 92.70%`
- 平均码率保持在约 `2Mbps`

这项优化我后面在面试里会重点讲，因为它是真正符合“发现问题 -> 找 sample -> 改参数 -> 录样本 -> 跑脚本 -> 出结果”这条闭环逻辑的。

## 6. 我怎么确认音频这块哪些能写，哪些不能写

音频也是我这次专门做过纠偏的一块。

### 6.1 当前真正做实的部分

我已经确认这些是当前工程真实完成的：

- AI 采集打通
- AENC AAC 编码打通
- 音频已进入 MP4
- 音频已进入 RTSP
- Windows 端 VLC 可以真实听到 RTSP 音频

### 6.2 我为什么没继续写 `THD < 5%`

我后来补了一个音频质量分析脚本 `analyze_audio_quality.py`，先拿现有 `wav/pcm` 样本把工具链跑通。

结果能正常输出：

- 采样率
- 声道数
- RMS
- 主频
- 粗略 THD

但我也发现一个很重要的边界：

- 当前样本的 `dominant_ratio` 很低
- 它不是标准单频正弦波
- 属于宽带/复杂音频

所以即使脚本算出了一个 THD 数字，我也不能把它当成正式简历指标去写。

也就是说，我这次没有为了让简历好看，强行把 `THD < 5%` 写成已实现，而是明确把它保留成“后续可补实验项”。
