# 性能类量化指标实验准备记录（2026-06-30）

## 1. 我为什么要补这份记录

旧材料里最容易让简历看起来很亮眼的一组数字，反而是我当前最不愿意直接照搬的一组：

- MP4 封装速度提升 `40%`
- I/O 调用次数下降 `60%`
- CPU 利用率变化

原因很直接：

- 当前这份工程没有现成的单线程 / 多线程对照实验报告
- 也没有一致输入下的 CPU / I/O / 吞吐统计结果

如果我把这些数字直接写进去，后面一旦被问“你怎么测的”，就会暴露出证据链不完整。

所以这次我先把这类指标推进到一个更诚实也更可执行的状态：

- **不乱写结果**
- **先把实验底座准备好**

## 2. 我当前已经确认到的事实

### 2.1 当前代码确实有可分析的路由日志

我检查了现有 `rtsp_module.c`，发现 router 线程已经会输出类似日志：

- `[ROUTER] ok=... packs=... pts=...`
- `[ROUTER] audio_ok=... last_len=... last_pts=...`
- `[ROUTER] exit, video_ok=... audio_ok=... fail=...`

这说明：

- 当前固件已经有一个天然的性能观察入口
- 我可以从日志里提取：
  - 视频包推进情况
  - 音频包推进情况
  - fail 次数
  - 基于 PTS 的粗略吞吐估计

### 2.2 我已经从真实历史运行日志里提取到第一批粗证据

这次我没有只停留在“代码里有日志接口”，而是把项目里已经留下来的 3 份真实控制台日志重新过了一遍：

- `old_qp_console.log`
- `qp_limited_console.log`
- `qp_baseline_timeout25_console.log`

这 3 份日志都明确出现了同一条并行链路启动信息：

- `ALL:1080p MP4(A/V)+RTSP(TCP:554 A/V)+VO(480x800) LCD`

这说明至少从运行态上看，项目当时确实是在尝试同时跑：

- LCD 预览
- MP4 音视频封装
- RTSP 音视频推流

我再用 `tools/analyze_router_log_metrics.py` 去解析这 3 份日志，得到一组一致的粗指标：

- 视频侧都有 `4` 个 router 采样点
- 从 `PTS` 反推的粗略视频推进速率都约为 `19.818 fps`
- 退出摘要里都能看到 `video_ok / audio_ok / fail`

其中两份带完整退出摘要的日志结果如下：

- `old_qp_console.log`
  - `video_ok = 443`
  - `audio_ok = 349`
  - `fail = 309`
  - `video_fail_ratio ≈ 0.4109`
- `qp_limited_console.log`
  - `video_ok = 445`
  - `audio_ok = 350`
  - `fail = 288`
  - `video_fail_ratio ≈ 0.3929`

这一步至少能支撑两个更稳妥的结论：

1. 当前架构不是纸面上的“支持三路/四路”，而是有真实运行日志证明它确实跑起来过。
2. 当前已经可以对 router 路由层的粗吞吐和 fail 情况做统一统计，而不是只能凭感觉判断“好像更稳了”。

### 2.3 当前音频侧还不能据此推出包率结论

我在解析日志时还发现一个很关键的细节：

- 日志里的 `audio_ok=... last_pts=0`
- 也就是当前历史日志没有留下可用的音频 `PTS`

这意味着：

- 视频侧可以先做粗略帧率估计
- 但音频侧还不能从这些旧日志里严谨反推出音频包率

所以现阶段我能说的是：

- 已经有视频推进和 fail ratio 的粗证据

但还不能说：

- 已经拿到完整的音视频双侧吞吐统计

### 2.4 当前还没有真正意义上的对照版本

我同时也确认了：

- 当前没有一个“明确保留的单线程基线版本”
- 所以我现在还不能严谨地说“性能提升 40%”

因为如果没有：

- 同样输入
- 同样运行时间
- 同样统计口径
- 两个结构版本对照

那这个百分比就站不住。

## 3. 我新增了什么工具

- `tools/analyze_router_log_metrics.py`

这个脚本可以直接解析固件日志中的 router 输出，给出：

- 视频日志样本数量
- 音频日志样本数量
- 基于 `PTS` 的视频粗略帧率估计
- 基于 `PTS` 的音频粗略包率估计
- 退出时 `video_ok / audio_ok / fail`
- `video_fail_ratio`

这一步的意义在于：

- 后面我要做性能实验时，不用再从头手工翻日志
- 可以统一统计口径

## 5. 后面如果要真正把 `+40% / -60%` 做实，我会怎么做

### 5.1 先保留一个对照版本

需要两份可运行版本：

1. 当前 router 版本
2. 一个更弱并发或更简单路由的基线版本

### 5.2 用统一输入和统一时长采样

比如：

- 固定运行 60s
- 固定同时开启 LCD / MP4 / RTSP / 音频
- 固定码率和分辨率

### 5.3 同时记录 3 组指标

1. router 日志指标  
   用 `analyze_router_log_metrics.py`

2. CPU 占用  
   用板端 `top` / `ps` / 运行期采样

3. I/O 或系统调用  
   如果环境允许，用 `strace` 或等价手段采样

只有这样，后面写：

- `+40%`
- `-60%`

这种数字才是站得住的。
