# IPC 项目面试深挖问答终稿（2026-06-30）

## 1. 项目一句话介绍

> 这是一个跑在全志 V853 上的 IPC 固件项目，我基于全志 MPP 打通了 `GC2053 -> VI -> LCD / VENC / MUX / RTSP` 主链路，同时把 `AI -> AENC -> AAC` 音频链路并进 MP4 和 RTSP，并且针对 LCD 偏色和 H.265 码流波动做了定位和优化。

### 回答

这几条链路是一起跑通的，不是各自单独写了个 demo。

## 4. 你项目里最有技术含量的问题是什么

### 回答

我觉得最能体现分析能力的是 LCD 预览偏色这个问题。

因为当时现象很容易误判成：

- ISP 坏了
- AE 有问题
- sensor 参数不对

但我后来发现：

- MP4 录制正常
- RTSP 视频正常
- 只有 LCD 预览异常

这说明全局 ISP 不是主要矛盾，问题更可能集中在 `VI2 -> VO` 这一条预览链路。

我后面做了三步：

1. 先把录像链路和预览链路拆开分析
2. 用诊断工具抓 `VI2` 原始帧
3. 对照 SDK sample 和 V4L2 色彩空间定义去定位像素格式/色彩空间问题

最后定位到关键点是：

- `VI2` 必须走 `NV12`
- `VI0` 录像链路走 `NV21`
- 预览链路里的 `color_space` 和窗口策略也要和 VO 预期一致

这不是靠反复试参数试出来的，而是通过拆链路、抓帧和对 sample 的理解定位出来的。

## 6. MP4 和 LCD 为什么会一个正常一个不正常

### 回答

因为它们虽然共用同一个 sensor 和 ISP 前端，但后面的分支不同。

这项目里：

- `VI0` 走录像/编码链路
  - 分辨率：`1920x1088`
  - 像素格式：`NV21`
  - 后面接 `VENC -> MUX / RTSP`
- `VI2` 走预览链路
  - 分辨率：`480x800`
  - 像素格式：`NV12`
  - 后面接 `VO -> LCD`

所以即使 ISP 前面是同一份 sensor 数据，后面每个 VI 通道的格式、窗口、颜色解释、输出目标都可以不一样。

这也是为什么：

- MP4 正常
- RTSP 正常
- LCD 不正常

并不矛盾。

## 7. 你怎么理解 VI0 和 VI2 为什么能拿到不同尺寸的数据

### 回答

我的理解是：

- Sensor 通过 MIPI CSI 把原始图像送进 SoC
- ISP 负责做去噪、AE、AWB、颜色校正这些图像处理
- 处理后的图像进入全志视频输入子系统
- 后面可以由不同的 VI/VIPP 通道分别取流

每个通道可以配置不同的：

- 输出尺寸
- 像素格式
- 目标下游模块

所以同一个前端图像源，完全可以：

- 一路给编码，保留接近原始的大分辨率
- 一路给 LCD，走适合显示的小分辨率

这也是硬件视频输入子系统做多路分发的意义。

## 8. 你 RTSP 这块是怎么做的

### 回答

RTSP 不是单独搞了一个“假 demo”，而是从现有编码链路里把真实码流路由出来。

最终结构更接近：

- `VENC/AENC` 出流
- `stream_router_thread` 拉流
- 同时把视频和音频分发给：
  - `MUX`
  - `RTSP`

也就是说：

- MP4 和 RTSP 不是两套完全独立的编码器
- 而是共用板端真实编码输出

视频这边：

- `AW_MPI_VENC_GetStream`
- 一份送 `AW_MPI_MUX_SendVideoStream`
- 一份送 `rtsp_push_h265`

音频这边：

- `AW_MPI_AENC_GetStream`
- 一份送 `AW_MPI_MUX_SendAudioStream`
- 一份送 `rtsp_push_aac`

这样才能保证：

- LCD
- MP4
- RTSP

三路同步工作，而不是互相打架。

## 10. 你做过哪些量化优化

### 回答

最完整的一项量化优化，是 H.265 码流稳定性优化。

我当时发现虽然工程里有 H.265 CBR 配置结构，但很多关键参数实际没有被稳定下发，日志里甚至出现过：

- `MaxQp=0`
- `MinQp=0`
- `VBV BufSize=0`
- `ThreshSize=0`

我对照全志 SDK sample 后，补了这几类参数：

- H.265 CBR 的 `QP` 范围
- `key_frame_interval`
- `VBV BufSize / ThreshSize`
- RC 层的源帧率 / 目标帧率

然后通过板端录制 MP4，再用离线脚本统计每帧大小离散系数 `CV`，最终结果是：

- `CV 143.70% -> 92.70%`
- 平均码率保持在约 `2Mbps`

这项是我项目里最完整的“改参数 -> 录样本 -> 跑脚本 -> 拿结果”闭环。
