# IPC 项目简历 Claim 严格核销清单（2026-06-30）

## 1. 这份清单为什么要单独存在

前面我已经做过：

- 量化项审计
- 简历纠偏
- 证据索引

但我继续往下深挖时，又发现一个更容易在面试里出问题的点：

- 有些内容不是“数字没测准”
- 而是**当前工程里根本没有实现那个子系统**

这种情况如果不单独标出来，后面最容易发生两种误讲：

1. 把“学长版本/方法论里的模块”说成“我当前代码里的模块”
2. 把“理论上可以形成的指标”说成“当前工程已经闭环的指标”

所以我补这份更严格的核销清单，目标不是润色，而是逐条判断：

- claim 从哪来
- 当前工程里到底有没有这项实现
- 现有证据是什么
- 我后面在简历/面试里应该怎么讲

## 3. 当前工程里已经确认与旧文档不一致的 claim

这些内容不是“没测”，而是**当前代码和旧文档已经明确冲突**。

- `沐一学长的项目梳理.md`
  - `44.1kHz`
  - `双声道`
  - `128kbps`

当前工程事实：

- `1ch`
- `16bit`
- `16kbps AAC`

当前证据：

- `src/main.c`
- `src/audio_module.c`
- `tools/collect_resume_metrics.py`
- `sample_mp4_full_verified_20260628.mp4`

结论：

- 这不是“口径差异”
- 而是当前工程和旧文档参数根本不同

面试/简历处理原则：

- 只能写当前工程的 `16kHz / mono / 16kbps`
- 不能再沿用 `44.1kHz / 双声道 / 128kbps`

## 5. 由“子系统不存在”直接推导出的结论

这一节很重要，因为它决定了某些数字为什么现在根本不能写。

### 5.1 `THD < 5%` 目前不只是“缺实验”，而是“当前链路范围也不匹配”

原因：

- 学长文档里的 `< 5%` 是按
  - 采集
  - 编码
  - 解码
  - 播放
  的全链路叠加在讲

但当前工程里：

- 有 `AI`
- 有 `AENC`
- 没有 `ADEC`
- 没有 `AO`
- 没有 `Lineout` 播放链路

这意味着：

- 当前工程可以做“采集/编码侧”的音频质量实验
- 但不能直接复述成“完整播出闭环 THD < 5%”

更准确的讲法：

- “我已经把 1kHz 正弦波 + FFT 分析工具链准备好了，后面可以测当前采集/编码链路的失真”

### 5.2 `AV sync < 30ms` 目前也不只是“数字没测”，而是“现有证据等级不够”

当前确实有：

- MP4 容器级时长对齐分析
- 轨道级粗同步结论

但当前缺：

- 每样本级或每帧级 `PTS` 精确对齐证据
- 当前历史音频路由日志里的有效音频 `PTS`

并且当前 router 历史日志里：

- `audio_ok=... last_pts=0`

所以结论只能到：

- “没有明显大漂移”

不能到：

- “同步偏差已经被证明 `< 30ms`”

### 5.3 `+40%`、`-60%` 目前不只是“没压测”，还夹杂架构代差

旧来源里讲的是：

- 生产者-消费者
- 双消费者并行
- 明确的单线程对照

当前工程实际结构更接近：

- `stream_router_thread`
- 直接拉 `VENC/AENC` 流
- 在 router 线程中扇出给 `MUX + RTSP`

这意味着：

- 当前确实是多线程流路由
- 但它不是学长文档里那套完全同构的实验对象

所以：

- 即使后面要做性能数字，也必须基于当前代码结构重新测
- 不能直接把旧版本的 `+40% / -60%` 挪过来
