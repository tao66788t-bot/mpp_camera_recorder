# 简历项目量化项纠偏与补证记录（第一人称版）

## 1. 背景

我在整理这段全志 V853 IPC 项目经历的时候，发现自己前期参考了很多学长的项目梳理、飞书格式文档和面试草稿。这样做虽然能快速搭骨架，但也带来了一个明显问题：

- 文档里出现了很多很漂亮的数字
- 但这些数字不一定都和我当前这份工程完全一致
- 有些是学长那一版的参数
- 有些是方法论上的推导值
- 有些甚至只是“应该这么测”的理论方案

如果我不先做一次系统审计，后面面试被追问到代码或实验细节时，很容易出现“简历写的是一套，当前代码实际跑的是另一套”的问题。

所以我专门花了一轮时间，把这份项目里所有和“量化结果”相关的内容重新梳理了一遍。

## 3. 我发现的核心问题

### 3.1 旧文档里的音频参数和当前工程不一致

这是我这次发现的第一个大冲突。

很多旧材料里写的是：

- `44.1kHz`
- `双声道`
- `AAC 128kbps`

但我回到当前代码里一看，真正跑起来的配置是：

- `16000Hz`
- `单声道`
- `16bit`
- `AAC 16kbps`

也就是说，如果我直接沿用旧文档里的音频数字，面试时一旦别人问到 `main.c` 或 `audio_module.c` 的具体配置，我立刻就会露馅。

这个问题提醒我：**项目整理不能只看旧文档，必须回到当前实际运行的代码。**

### 3.2 有些“效果提升”类数字更像学长模板，不是我当前工程的实测值

我重点盯了几组最容易被写进简历的数字：

- 色彩还原度提升 `20%`
- 码流稳定性提升 `30%`
- 音频总失真率 `< 5%`
- MP4 封装速度提升 `40%`
- I/O 调用次数下降 `60%`
- 音视频同步偏差 `< 30ms`

审完之后我发现：

- `码流稳定性` 这项，我当前工程里确实有真实闭环
- 其他几项大多还停留在旧文档说法、理论推导或方法设计层面

也就是说，这些数字不是完全不能做，而是**当前这份工程还没有把它们全部做成“我自己可以拿证据证明”的结果。**

### 3.3 当前 MP4 / RTSP 路由结构和旧文档的“完整生产者消费者模型”也不完全一样

旧文档里经常用非常完整的生产者-消费者话术去讲：

- 一个 producer
- 两个 consumer
- 共享节点
- 双队列
- 引用计数

我回头看当前代码发现：

- 现在确实是多线程
- 但核心路径更接近 `RTSP server thread + stream_router_thread`
- `stream_router_thread` 里直接做 `VENC_GetStream / AENC_GetStream`
- 然后同步把数据扇出给 `MUX` 和 `RTSP`

所以我现在能讲的是：

- “我已经把多线程码流路由跑通了”
- “我理解共享节点 / 双队列的设计”

但不能把当前代码硬说成已经是完整的“双消费者共享节点引用计数实现”。

## 4. 当前真正能讲实的数字

这轮梳理之后，我把“当前最安全、最能自证”的量化项收敛成下面几条。

### 4.1 视频主链路

- LCD 本地预览：`480x800`
- 编码链路：`1920x1088 @ 20fps`
- RTSP 服务端口：`554`

这些都能从当前 `main.c` 和现有验证文档里直接找到证据。

### 4.2 音频链路

- 音频采样率：`16000Hz`
- 位宽：`16bit`
- 声道：`单声道`
- AAC 码率：`16kbps`

这部分我已经通过代码和 MP4 验证文档对齐过了，能讲得很稳。

### 4.3 码流波动优化

这一项目前是我手里最完整的“前后对照型指标”：

- `cv_percent = 143.70 -> 92.70`
- 目标码率保持在约 `2Mbps`

并且我知道这项是怎么做出来的：

- 调整 H.265 CBR 的 QP 范围
- 补齐 VBV 参数
- 补齐 RC 层帧率信息
- 板端录制 MP4
- 用离线脚本统计帧大小离散程度

这条在面试里最值得重点讲。

### 4.4 端侧检测

- 人脸模型：`Facedet_480_288_nv12.nb`
- 输入尺寸：`480x288`
- 检测阈值：`0.60`

这部分也已经从“代码里留了接口”升级成了“板端真实跑通”。

## 5. 我决定怎么纠偏简历内容

### 5.1 不再直接照抄旧模板

这次最大的纠偏点就是：

- 我不会再直接照抄学长文档里的量化项
- 也不会为了让简历更漂亮，就把还没实测的数字直接写成结果

以后我只保留两类数字：

- 当前工程已经有代码和实验支撑的
- 我后面能补出完整实验闭环的

### 5.2 先保住当前最稳的数字

短期内我准备优先保留：

- `480x800 LCD`
- `1920x1088@20fps`
- `RTSP 554`
- `16kHz/16bit/单声道 AAC`
- `CV 143.70% -> 92.70%`
- `480x288` 人脸检测输入

这些是当前项目里最不容易被追问击穿的部分。

### 5.3 后面再补真正值得写进去的实验

我后面准备按优先级补下面几项：

1. 把当前 MP4 / 音频 / CV 信息做成自动汇总脚本  
   这样所有当前真实能力都能一键出结果。

2. 补色彩还原度 `20%` 的实测闭环  
   用色卡 + ΔE，而不是只写“色彩更好了”。

3. 补多线程封装性能实验  
   真正做单线程 / 多线程对照，再决定能不能写 `+40%`。

4. 补音频 THD 或总失真闭环  
   这个优先级最低，因为外部实验条件要求更高。

## 6. 这次纠偏给我的经验

这轮工作让我真正意识到：

- “会讲项目”不等于“项目讲得住”
- 文档写得再完整，也不能代替当前工程真实状态
- 只要涉及百分比、码率、失真率、同步误差、性能提升这类数字，就必须有对应证据

对我来说，这次最有价值的不是又多写了一份总结，而是建立了一个以后整理项目经历的原则：

> 任何量化项，必须能回到代码、日志、样本文件、分析脚本或实验记录里找到出处。

只有这样，后面面试时我讲出来的每一个数字，才更像是“我自己真的做出来的结果”。
