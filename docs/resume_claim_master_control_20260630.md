## 3. 当前绝对不能写死的内容

这些内容不是“听起来不稳”，而是当前证据明确不足，或者已经和当前工程事实冲突。

| 条目 | 当前状态 | 为什么不能直接写 |
|------|------|------|
| `44.1kHz / 双声道 / 128kbps` | 与当前工程冲突 | 当前代码和 MP4 样本都显示真实音频参数是 `16kHz / mono / 16kbps AAC` |
| `Opus -> ADEC -> AO / Lineout` | 当前工程不存在 | 代码里没有完整 `Opus/ADEC/AO` 主线 |
| `AEC / ANS / AGC / SetDevVolume` 已生效 | 当前工程未发现 | 代码检索未发现当前主线里有这些调用闭环 |
| `THD < 5%` | 仅实验准备完成 | 还没有拿当前板端链路做标准信号闭环测量 |
| `色彩还原提升 20%` | 仅实验准备完成 | 有 ΔE 工具链，但没有真实色卡 before/after 证据 |
| `AV sync < 30ms` | 只有粗证据 | 现在只有封装轨道级时长差，不是事件级同步测量 |
| `封装速度 +40%` | 只有粗证据 | 当前架构与学长旧文档不同，不能直接照搬 |
| `I/O 调用次数 -60%` | 只有粗证据 | 缺统一条件下的 I/O 采样和前后对照 |

## 6. 如果我要继续把剩余数字做实，顺序应该怎么排

建议顺序：

1. `THD < 5%`
2. `色彩提升 20%`
3. `AV sync < 30ms`
4. `封装速度 +40% / I/O -60%`

原因：

- `THD` 和 `ΔE` 最容易补成标准化实验
- `AV sync` 需要事件设计，但仍然可控
- 性能类数字最容易被“测试条件不统一”污染，应该放最后
